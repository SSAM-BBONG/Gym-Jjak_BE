# Chatbot Internal Tool API

- Created: 2026-07-23
- Last updated: 2026-07-23
- Base path: `/internal/chatbot/tools`

FastAPI LangGraph Function Calling only uses these APIs to fetch evidence for an answer. The response is not delivered to the frontend directly. FastAPI uses it to compose the final answer, then the existing `done` event is relayed through Spring WebSocket/STOMP.

The workout-history API is the current source of a member's routine record: Gym-Jjak stores the exercise, body part, and sets selected by a member in workout diaries.

## Authentication and actor binding

Every request needs these headers.

| Header | Purpose |
| --- | --- |
| `X-Internal-Api-Key` | Authenticates the FastAPI server. It must equal `AI_INTERNAL_API_KEY`. |
| `X-Chatbot-Session-Id` | The chatbot session ID generated by Spring. |
| `X-Request-ID` | The active stream request ID generated by Spring. |

The API never accepts `userId` as a parameter. Spring finds the session with `X-Chatbot-Session-Id`, verifies the matching active request is not expired, and uses the session owner as the query user. This blocks a tool argument generated by an LLM from changing the target user.

This restriction applies to both sides of the internal tool contract: Spring does not expose a `userId` request field, and the FastAPI LangGraph tool schema must not define `user_id`. A valid tool signature is `get_workout_history(from, to)`.

The initial Spring-to-FastAPI chatbot request may still contain `actor.user_id`. That value is server-created context from the authenticated STOMP Principal, not an LLM tool argument. FastAPI must propagate `session_id` and `X-Request-ID` as headers when it calls this API instead of passing `actor.user_id` as a parameter.

## Latest inbody

`GET /internal/chatbot/tools/inbody/latest`

```json
{
  "data": {
    "measuredDate": "2026-07-23",
    "weight": 70.0,
    "bodyFatPercentage": 20.0,
    "skeletalMuscleMass": 30.0
  }
}
```

`data` is `null` when the user has no inbody record.

## Workout history

`GET /internal/chatbot/tools/workout-history?from=2026-07-01&to=2026-07-23`

The period must be between one and 31 days. The result only contains the date, exercise, body part, and number of sets needed for an answer.

```json
{
  "data": {
    "from": "2026-07-01",
    "to": "2026-07-23",
    "diaries": [
      { "date": "2026-07-23", "exercise": "squat", "part": "LEG", "setCount": 4 }
    ]
  }
}
```

## Error policy

- Invalid server API key: `401 Unauthorized`
- Session/request ID mismatch or expired stream: `CHATBOT_TOOL_403_1`
- Invalid date range: `CHATBOT_TOOL_400_1`
- Invalid date format: Spring common error response with HTTP `400`

`CHATBOT_TOOL_403_1` and `CHATBOT_TOOL_400_1` are returned by the global Spring error handler in its standard error envelope (`timestamp`, `status`, `code`, `message`, `traceId`, `details`). The API-key filter rejects unauthenticated server calls before the controller is invoked.

## FastAPI implementation rule

1. Register `get_latest_inbody()` and `get_workout_history(from, to)` as read-only LangGraph tools.
2. Send `X-Internal-Api-Key`, `X-Chatbot-Session-Id`, and `X-Request-ID` on every tool HTTP request.
3. Put the response into a `ToolMessage`; do not expose the API key, session owner ID, or raw internal headers to the model.
4. Spring's chatbot package continues to use `Chatbot...QueryPort → Adapter → domain UseCase`, never another domain's Repository or JPA entity directly.
